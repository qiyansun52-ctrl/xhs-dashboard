# UI 视觉打磨 + Agent 工作台设计文档

## 概述

对小红书团队管理平台做一次整体视觉升级。方向是 **A（精炼打磨）+ B（高级暗色）的融合**：保留现有深色美学与信息架构，引入 glass 质感、品牌红渐变光晕、统一的间距与字体层次、一套微动效规范。目标是让 4-8 名非技术成员在不改变使用习惯的前提下，明显感知到产品「更精致、更高级」。

本文档主体（第 1–10 节）定义**视觉与交互规范**。结构性改动（详情抽屉化、AI 双 Tab 等）单列在第 8.2 节，标注为「需单独决策」，不在首期实现范围内。

第 11 节定义 **Agent 工作台**——把现有的 AI 搜索框升级为真正能产出运营成果的运营 Agent（UI/UX 层 + 首期功能优先级）；其能力架构与策略边界以 `docs/superpowers/specs/2026-05-10-xhs-operating-agent-design.md` 为准。实现计划另见 `docs/superpowers/plans/` 下的后续 plan 文件。

## 1. 设计方向决策

设计探索阶段提供了三个方向，最终结论：

| 方向 | 决策 | 说明 |
|------|------|------|
| A · 精炼打磨 | 采纳（基线） | 修间距 / 字体层次 / hover 态，加微动效，风险最低 |
| B · 高级暗色 | 采纳（叠加） | glass morphism、渐变光晕、品牌红更饱和，视觉冲击最强 |
| C · 简洁克制 | 不采纳 | 扁平化与本产品的「高级感」目标冲突 |
| 桌面侧栏 | C · 展开 + 光晕 | 展开文字栏 + 激活项红色光晕 + 玻璃质感卡片 |

最终落地 = **A+B 视觉语言 + 展开光晕侧栏**。

## 2. 设计目标与原则

1. **非技术用户友好**：导航始终带中文标签，不靠图标猜功能；状态变化要有明确反馈。
2. **改动可控**：以现有组件为基础叠加样式，不重写页面结构，不删除已有功能（见 CLAUDE.md「重要约束」）。
3. **一致性优先**：所有卡片、按钮、间距、动效走统一 token，不再各页面各写一套。
4. **性能不退化**：动效以 transform / opacity 为主，避免触发重排；骨架屏与 count-up 不阻塞首屏。
5. **移动端等价**：每条桌面规范都要给出移动端对应表现，手机底部 tab 栏结构保持不变。

## 3. 设计系统基础

由于项目约定**全部 inline styles、不引入 CSS 框架**，下列 token 以「JS 常量对象」形式落地，建议集中在 `src/shared.jsx` 导出，各页面引用。

### 3.1 色彩体系

| 用途 | 现值 | 新值 | 说明 |
|------|------|------|------|
| 页面背景 | `#0a0a0a` | `linear-gradient(135deg,#0a0a0a,#0f0f18)` | 轻微蓝黑偏移，几乎不可察觉但拉开层次 |
| 侧边栏背景 | `#0e0e0e` | `linear-gradient(180deg,#0f0f1a,#0a0a12)` | 顶部略亮，底部更深 |
| 卡片背景 | `#111` | `rgba(255,255,255,0.03)` | glass 质感，依赖背景透出 |
| 卡片边框 | `#1e1e1e` | `rgba(255,255,255,0.07)` | 半透明细边框 |
| 表单边框 | `#2a2a2a` | `rgba(255,255,255,0.1)` | 比卡片边框略明显 |
| 品牌主色 | `#FF2442` | `#FF2442`（纯色保留） | 强调块 / 渐变起点用 `linear-gradient(135deg,#FF2442,#ff6b6b)` |
| 品牌光晕 | — | `0 0 8px rgba(255,36,66,0.35)` | 头像、激活项、主按钮 |
| 成功绿 | `#26DE81` | `#26DE81` | 不变 |
| 警告橙 | `#FF9F43` | `#FF9F43` | 不变 |
| 文字主色 | `#e0e0e0` | `rgba(255,255,255,0.88)` | 略提亮 |
| 文字次要 | `#666` / `#555` | `rgba(255,255,255,0.45)` / `0.3` | — |

> 现有纯色（`#111` / `#1e1e1e` 等）不强制全量替换，新写与改动的卡片走新值即可；同一页面内不应新旧混用。

### 3.2 间距

统一间距单位：**4 / 8 / 12 / 16 / 24 / 32 px**。卡片内边距默认 `14px`，卡片间距 `12px`，页面边距桌面 `20px` / 手机 `12px`（沿用现状）。

### 3.3 字体层次

字体仍为 DM Sans。四级层次：

| 层级 | 字号 | 字重 | 颜色 |
|------|------|------|------|
| 页面标题 | 24 | 600 | `#fff` |
| 卡片标题 | 16 | 500 | `rgba(255,255,255,0.88)` |
| 正文内容 | 13 | 400 | `rgba(255,255,255,0.7)` |
| 辅助标签 | 11 | 400/600 | `rgba(255,255,255,0.45)` |

### 3.4 圆角与阴影

- 圆角：卡片 `12px` · 按钮 `8px` · 标签/pill `6px` · 头像 `50%`
- 卡片阴影（统一）：`0 1px 0 rgba(255,255,255,0.04) inset, 0 4px 16px rgba(0,0,0,0.3)`
  - inset 高光制造「玻璃」边缘，外阴影制造浮起感。
- 主按钮阴影：`0 2px 8px rgba(255,36,66,0.35)`

### 3.5 动效 token

| token | 值 |
|------|------|
| 卡片 hover 过渡 | `200ms ease` |
| 按钮点击过渡 | `100ms ease` |
| 弹窗出现 | `220ms` |
| 页面切换 | `150ms` |
| count-up 数字 | `600ms` |

## 4. 导航壳层

### 4.1 桌面侧栏（展开 + 光晕）

现状 `App.jsx` 桌面侧栏已是 230px 展开式、图标 + 中文标签常驻。本次**保留 230px 宽度与现有结构**（Logo / 导航 / 账号列表 / 团队成员），仅叠加视觉：

> 设计探索中的 140px 侧栏 mockup 仅作风格示意。实际侧栏需容纳账号列表与成员列表，保持 230px。

- 背景改为 `linear-gradient(180deg,#0f0f1a,#0a0a12)`，右边框 `1px solid rgba(255,36,66,0.08)`。
- **激活导航项**：背景 `rgba(255,36,66,0.1)`，边框 `1px solid rgba(255,36,66,0.25)`，阴影 `0 0 16px rgba(255,36,66,0.12) inset`；图标染品牌红并带 `0 0 6px rgba(255,36,66,0.6)` 光晕。
- **hover 导航项**：背景 `rgba(255,255,255,0.03)`，文字提亮到 `0.6` 透明度。
- Logo 方块改为品牌红渐变 + 光晕。
- 账号列表项 hover 加 `rgba(255,255,255,0.03)` 底色（当前无 hover 态）。

### 4.2 移动端

手机顶部 bar 与底部 tab 栏**结构不变**。仅微调：

- 底部 tab 激活项：图标与文字染品牌红（现状已有），额外在图标下方加一个 `3px` 红色圆点指示当前页。
- 顶部 Logo 方块同步改为渐变。
- 底部安全区 `env(safe-area-inset-bottom)` 保持。

## 5. 卡片组件升级规范

所有卡片（账号卡片、帖子卡片、统计卡片、素材卡片、AI 来源卡片等）统一为「glass 卡片」：

```
background: rgba(255,255,255,0.03)
border: 1px solid rgba(255,255,255,0.07)
border-radius: 12px
box-shadow: 0 1px 0 rgba(255,255,255,0.04) inset, 0 4px 16px rgba(0,0,0,0.3)
padding: 14px
```

**hover 态**（仅可点击卡片）：`border` 提亮到 `rgba(255,255,255,0.14)`，`transform: translateY(-1px)`，过渡 `200ms ease`。

建议在 `shared.jsx` 增加一个 `<Card>` 原子组件或 `cardStyle` 常量，逐步替换各页面手写的 `background:"#111"` 卡片。替换是渐进式的，不要求一次性全改。

## 6. 微动效与交互规范

| 场景 | 表现 |
|------|------|
| 卡片 hover | `translateY(-1px)` + border 提亮，`200ms ease` |
| 按钮点击 | `scale(0.97)`，`100ms ease`（`:active`，用 `onMouseDown/Up` 或注入 class） |
| 弹窗出现 | `fadeIn` + `slideUp 8px`，`220ms` |
| 页面切换 | 内容区 `fadeIn`，`150ms` |
| 数字变化 | count-up 动画，`600ms`（统计卡片、汇总数字） |

### 6.1 keyframes 落地方式

`@keyframes` 无法用 inline style 表达。项目已有先例：`App.jsx:169` 的单个全局 `<style>` 块（`@keyframes spin`）。

**规范**：所有新增 keyframes（`fadeIn` / `slideUp` / `shimmer` / `countUp` / `glowPulse` 等）一律追加到这同一个 `<style>` 块。这不违反「全部 inline styles」约束——该约束针对的是组件样式，全局动画关键帧是公认例外。不要为此引入 CSS 文件或框架。

count-up 数字建议用 JS（`requestAnimationFrame` 计数）实现而非纯 CSS，便于绑定真实数值。

## 7. 状态反馈规范

| 状态 | 表现 |
|------|------|
| Loading | 骨架屏（`shimmer` 流光动画）替代 spinner；spinner 仅保留在按钮内 |
| 保存成功 | 绿色 toast，右上角滑入，2s 自动消失 |
| 报错 | 红色 toast + 中文错误内容（错误提示必须中文，见 CLAUDE.md） |
| 空状态 | 插图 / 大图标 + 引导文字 + 一个操作按钮，不留纯空白 |
| 按钮提交中 | 按钮禁用 + 内嵌旋转图标，防重复提交 |

toast 建议做成全局组件（`shared.jsx` 导出 `useToast()`），避免各页面各写一套。骨架屏建议提供 `<Skeleton>` 原子组件。

## 8. 六个页面改造重点

### 8.1 首期范围（纯视觉打磨）

不改数据流、不改组件结构，只换样式与加动效：

| 页面 | 改造点 |
|------|--------|
| 账号管理 | 账号卡片 → glass 质感 + 头像光晕；统计数字加 count-up；折线图换 area chart + 渐变填充 |
| 内容管理 | 帖子卡片封面图优先展示（有图用图，无图显标题）；状态 badge 加色块填充；筛选栏改横向滚动 pill tabs |
| 日历 | 日期格子 hover 显示帖子 tooltip；当天用红色光晕圆圈高亮；有帖子的日期底部加账号色小点；月视图顶部加「本月待发 X 条」汇总 |
| 素材库 | Tab 栏改下划线激活态 + 滑块动画；每个 tab 顶部加搜索框 + 常用筛选；对标账号卡片加头像 + 平台 icon；标题库加「一键复制」按钮 |
| AI 助手 | 搜索框置顶放大为主角；证据质量条从小 badge 升级为横向进度条 + 图标；来源卡片加缩略图区；研究笔记改时间线 feed |
| 数据分析 | 顶部 4 个汇总数字卡片加 count-up；图表区加时间范围选择器（7d/30d/90d）；账号折线图颜色跟随账号主色；空数据显示引导插图 |

### 8.2 需单独决策（结构性改动，不在首期）

以下改动会改变组件结构或导航状态，与「不删除已有功能」约束需逐项权衡，**首期不实现**，待产品确认后单独开 plan：

- **账号详情抽屉化**：用右侧 Drawer 替代当前 `AccountDetail` 内嵌展开。影响 `AccountsPage` 交互流程。
- **AI 搜索 / Agent 拆为并列 Tab**：现为页面内 `aiMode` 嵌套切换；拆分会新增一个顶级导航项并改动 `view` 状态结构。
- **新建帖子弹窗分步骤化**：基本信息 → 图片 → 排期 三步流程，重构现有单页弹窗。

> 这些项各自价值明确，但属于交互重构而非视觉打磨。建议视觉打磨上线、团队适应后再评估。

## 9. 实现约束

强约束（违反需重新评审）：

- **不用 TypeScript**，保持纯 JavaScript。
- **不用 CSS 框架**，全部 inline styles；唯一例外是 `App.jsx` 单个全局 `<style>` 块内的 `@keyframes`。
- **不用路由库**，`useState` 控制视图（首期不动）。
- **不删除已有功能**，所有改动在现有基础上叠加。
- 错误提示必须中文。
- 设计 token 集中在 `src/shared.jsx`，不散落各页面。
- 改动渐进式：允许新旧卡片样式在不同页面共存，但**同一页面内不得新旧混用**。

## 11. Agent 工作台（真正的运营 Agent）

### 11.1 定位与现状

现状：`AgentPage` 已有 agent runtime 雏形（`agent_runs` / `agent_steps` / `tool_invocations` 三张表，commit `fc41133`），但目前只跑通「研究问答」一条链路——对非技术用户来说，它仍然像一个 AI 搜索框，不像一个能干活的助手。

本节的目标是把它升级为**真正能产出运营成果的 Agent 工作台**：用户用一句话说目标，Agent 自己规划 → 调用工具 → 观察 → 决策 → 产出**可直接落地的成果**（周排期、文案草稿、数据周报、爆款模板），所有写库 / 发布类动作一律走人工确认。

> Agent 的能力架构、工具注册表、证据门、半自动边界以 `docs/superpowers/specs/2026-05-10-xhs-operating-agent-design.md` 为准（9 个专家技能、`Plan → Tool → Observe → Decide → Answer → Review → Memory` 闭环）。本节是它的 **UI/UX 层 + 首期功能优先级**，不重复策略细节。

### 11.2 工作台布局

桌面三栏，沿用 glass 视觉语言：

```text
┌─────────────┬──────────────────────────┬──────────────┐
│ 左栏 上下文   │ 中栏 对话与产出            │ 右栏 过程与确认 │
│ · 目标输入框  │ · Agent 产出卡片           │ · 任务流 timeline│
│   （大、置顶）│   计划/草稿/报告/拆解       │ · 证据面板      │
│ · 上下文 chip │ · 每张产出卡带「落地」按钮   │ · 待确认动作    │
│   账号/国家/  │   （进日历/存草稿/存模板）   │                │
│   时间范围    │                          │                │
│ · 图片上传    │                          │                │
│ · 历史 run    │                          │                │
└─────────────┴──────────────────────────┴──────────────┘
```

手机端单列：目标输入置顶 → 产出居中 → 任务流 timeline 与待确认队列折叠为可展开的底部 sheet（沿用全站弹窗规范）。

### 11.3 任务流可视化

让 Agent「可信」而不是「魔法」——用户必须能看见它在做什么。

中栏产出旁 / 右栏放一条**实时 timeline**，每个 step 一张小卡片，左侧竖线连接：

| step | 图标 | 文案示例 |
|------|------|----------|
| 规划 | 🎯 | 已理解任务：周内容规划 / 英国账号 / 下周 |
| 调用工具 | 🔧 | 正在检索素材库… |
| 观察 | 👁 | 找到 6 条强匹配素材 |
| 决策 | ⚖️ | 内部证据充足，无需外部发现 |
| 产出 | 📄 | 已生成 7 天排期草案 |
| 待确认 | ⏸ | 等待你确认是否写入日历 |

step 状态：运行中（红色脉冲光晕 `glowPulse`）/ 完成（绿色对勾）/ 失败（红色，附中文错误）。timeline 数据直接来自 `agent_steps` 表，保证可追溯。

### 11.4 待确认队列（human-in-the-loop）

写动作不自动执行，统一进入待确认队列。

- 工作台顶部常驻一个 badge：`待确认 N`，N>0 时品牌红、带轻微脉冲。
- 点开是审核卡片列表，每张卡片显示：动作类型、产出预览、**Agent 为什么这么做**、证据分、重复警告、`通过 / 拒绝` 按钮。
- 拒绝走结构化原因：`不相关 / 低质量 / 疑似广告 / 重复素材 / 不符团队调性 / 数据异常 / 已入库`。
- 数据落 `agent_review_actions` 表（见 agent 设计文档 6.1）。

### 11.5 首期落地的「真正能用」功能

以下 8 个功能按团队（海外留学生人设账号运营）的真实高频痛点排序，建议作为首期实现集。每个功能 Agent 都产出一个**可落地的成果物**，而不仅是聊天回复：

| 功能 | 用户怎么说 | Agent 做什么 | 产出形态 | 落地动作（需确认） |
|------|-----------|-------------|----------|-------------------|
| 周内容规划 | 「帮我排下周英国账号的内容」 | 读账号人设 + 日历空档 + 素材库 + 历史表现，规划 7 天选题 | 周计划表（日期 × 账号 × 选题） | 一键写入日历、分配负责人 |
| 爆款拆解 | 贴小红书链接 / 选候选帖 | 抓笔记 + 评论，拆标题钩子 / 开头 / 结构 / 配图 / 情绪点 | 拆解卡片 | 存为团队模板 |
| 按人设写文案 | 「用 Jasper_Page 的口吻给这个选题写 5 个标题」 | 检索人设 + 素材，生成标题 / 正文 / tag 多版本 | 文案包 | 存草稿、进日历 |
| 发布前风险审核 | 贴一段草稿文案 | 查违禁词 + 夸大宣传 + 广告口吻 + 人设冲突 | 风险报告（标红 + 改写建议） | 仅给建议，发布决策在人 |
| 评论区洞察 | 选一篇已发布帖子 | 拉评论，聚类痛点 / 疑问 / FAQ | 洞察摘要 + 追更选题 | 存洞察到团队记忆 |
| 选题灵感 | 「给我几个英国春天能出收藏的选题」 | 检索素材库 + 历史 + 趋势 | 选题清单（含理由 + 证据） | 加入选题库 / 日历 |
| 对标账号监控（定时） | 每周自动跑 | 扫 benchmark 账号新爆款，按收藏增速排序 | 爆款告警 digest | 候选入素材库 |
| 周数据复盘（定时） | 每周一自动跑 | 聚合上周帖子 / 账号数据 | 数据周报（增长洞察 + 建议） | 采纳计划调整 |

前 6 项是用户主动发起；后 2 项是定时主动任务（见 11.6）。每项对应 agent 设计文档第 3 节的专家技能，不新增能力，只是把它做成有明确成果物的端到端流程。

### 11.6 主动 Agent：待办与提醒

让 Agent「主动」而不只是「被问」——这是它从工具变成助手的关键。

- 工作台首屏顶部增加「待办」区，聚合两类卡片：①定时任务（监控 / 周复盘）的最新产出；②待确认队列中的高优先项。
- 对标监控发现新爆款、周复盘生成周报时，在「待办」区出现一张卡片，可选配站内提示。
- 定时任务复用爬虫已有的 GitHub Actions 调度思路；产出仍走待确认，不自动入库。
- 「待办」为空时显示空状态插图 + 引导文案，不留空白。

### 11.7 视觉与交互规范

Agent 工作台完全沿用本文档第 3–7 节：glass 卡片、品牌红光晕、统一间距字体、微动效、骨架屏 / toast / 空状态。新增两条 keyframes（追加到 `App.jsx` 全局 `<style>` 块）：

- `glowPulse`：运行中 timeline step 的红色呼吸光晕，`1.6s ease-in-out infinite`。
- `timelineEnter`：新 step 卡片入场，`fadeIn + slideUp 6px`，`200ms`。

产出卡片（计划 / 草稿 / 报告 / 拆解）= glass 卡片 + hover 浮起；卡片内「落地」按钮用主按钮样式（渐变 + 光晕）。

### 11.8 实现分期

对齐 agent 设计文档第 8 节的 Phase 划分，本节只补充 UI 侧增量：

| 阶段 | 内容 |
|------|------|
| A | 工作台三栏布局 + 实时 timeline（消费已有 `agent_steps`），让现有研究链路「看得见过程」 |
| B | 待确认队列 UI + `agent_review_actions` 接入；产出卡片「落地」按钮 |
| C | 逐个上线 11.5 的主动发起功能（周规划 → 爆款拆解 → 写文案 → 风险审核 → 评论洞察 → 选题灵感） |
| D | 定时任务 + 「待办」区（对标监控、周复盘） |

> 强约束不变：写 / 发布类动作一律人工确认；外部抓取内容在审核入库前始终标记为「未审核候选」；不做自动发布 / 评论 / 点赞。

> **前置依赖**：Phase B 实施前须先解决第 12 节第 5 项（`agent_review_actions` 建表）；Phase C 的爆款拆解 / 评论洞察须先解决第 6 项（爬虫适配器）；Phase D 须先解决第 7 项（定时调度方式）。这些不解决，对应阶段无法开工。

## 12. 待决策事项

1. 侧栏宽度确认保持 230px（本文档假设保持）。
